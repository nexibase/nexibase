# i18n Phase 2 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Phase 1에서 미룬 모든 영역의 하드코딩된 한국어 텍스트를 영어/한국어 i18n으로 전면 전환한다 (약 1,100~1,300개 문자열).

**Architecture:**
- Phase 1 인프라(next-intl, `[locale]` 라우트, `src/messages/{en,ko}.json` 머지) 위에서 작업
- 영역별로 plugin locale 파일을 추가/확장 (`src/plugins/{shop,contents,policies}/locales/`) 또는 코어 locale 확장 (`src/locales/`)
- 클라이언트 컴포넌트는 `useTranslations`, 서버 컴포넌트/API는 `getTranslations`
- 각 sprint(=Task 묶음)는 독립적으로 빌드 가능 + 즉시 배포 가능

**Tech Stack:** next-intl, Next.js 16 App Router, TypeScript

**Phase 2의 구조:**
이 문서는 **9개 Sprint**로 나뉘며 각 Sprint는 독립 Task 입니다. Sprint 단위로 sub-agent에 전달하면 효율적입니다. 각 Sprint 끝에 빌드 검증 + 커밋이 포함됩니다.

| Sprint | 영역 | 예상 문자열 | 우선순위 |
|--------|------|-------------|----------|
| 1 | 코어 locale 키 확장 + 플러그인 locale 파일 생성 | (인프라) | 필수 (선행) |
| 2 | 코어 인증 페이지 (login/signup/email-certify/verify-email) | ~80 | 높음 |

| 5 | Boards 플러그인 — Post/Write/Edit/CommentReactions + admin | ~150 | 높음 |
| 6 | Shop 플러그인 — 사용자 측 (product/cart/checkout/order/mypage) | ~250 | 중요 |

**Phase 2 완료 기준:**
- `/en/...` 경로의 모든 페이지에서 한국어 텍스트가 0건 (DB에서 오는 동적 데이터 제외)
- `/ko/...` 경로의 모든 페이지에서 영어 텍스트가 0건 (DB에서 오는 동적 데이터 제외)
- `npm run build` 통과
- 핵심 API 에러 메시지(auth/payment/order)가 locale에 따라 응답
- 비핵심 API 에러 메시지(notification/admin internal/etc.)는 한국어 유지(Phase 3로 deferred)

## 사전 작업 패턴 (모든 Sprint 공통)

### A. Plugin locale 파일 생성 패턴

플러그인이 첫 번째로 i18n을 추가할 때:
```bash
mkdir -p src/plugins/{plugin}/locales
```
그리고 `en.json`, `ko.json` 두 파일을 만들고 **반드시 unique top-level namespace**를 사용 (예: `shop`, `contents`, `policies`). 다른 플러그인 또는 코어 namespace와 충돌하면 안 됨 (scan-plugins의 `mergeLocales`는 `Object.assign` 얕은 머지를 사용).

### B. scan-plugins 재실행

locale 파일을 추가/수정한 뒤에는 항상 다음을 실행:
```bash
node scripts/scan-plugins.js
```
이것이 `src/messages/{en,ko}.json`을 다시 머지함. 머지 결과를 안 보이면 next-intl이 키를 못 찾음.

### C. 클라이언트 vs 서버 컴포넌트

- 파일 첫 줄에 `"use client"`가 있으면 → `useTranslations` 사용
  ```tsx
  "use client"
  import { useTranslations } from 'next-intl'
  
  export default function Page() {
    const t = useTranslations('namespace')
    return <h1>{t('title')}</h1>
  }
  ```
- 서버 컴포넌트(`"use client"` 없음)나 API 라우트면 → `getTranslations` 사용
  ```tsx
  import { getTranslations } from 'next-intl/server'
  
  export default async function Page() {
    const t = await getTranslations('namespace')
    return <h1>{t('title')}</h1>
  }
  ```
- API 라우트 예:
  ```ts
  import { getTranslations } from 'next-intl/server'
  
  export async function POST(request: NextRequest) {
    const t = await getTranslations('errors')
    if (!user) {
      return NextResponse.json({ error: t('loginRequired') }, { status: 401 })
    }
  }
  ```
  next-intl은 요청에서 locale을 자동 추출 (URL의 `[locale]` 세그먼트가 있을 때). API는 `[locale]` prefix가 없으므로 명시적으로 locale을 전달해야 할 수 있음 — 자세한 처리는 Sprint 9 참고.

### D. 빌드 + 검증

각 Sprint 끝:
```bash
npx next build
```
빌드 통과해야 commit. 빌드 실패 시 누락된 키, 잘못된 namespace, 또는 type 오류 — 그 자리에서 수정.

### E. Korean residual 검색

특정 파일/디렉터리에 한국어가 남아있는지 확인:
```bash
grep -rn '[가-힣]' src/path/ | grep -v '//' | grep -v '/\*' | head -50
```
완벽한 검출은 아니지만(주석을 일부 놓침) 90% 케이스 커버.

`src/proxy.ts`의 `'이 기능은 비활성화되었습니다.'` 메시지를 locale에 따라 응답하도록 처리. 미들웨어에서는 `getTranslations`가 직접 동작하지 않으므로 경로에서 locale을 추출하여 직접 메시지 매핑:

```ts
const DISABLED_MESSAGES: Record<string, string> = {
  en: 'This feature is disabled.',
  ko: '이 기능은 비활성화되었습니다.',
}

function disabledMessage(pathname: string): string {
  for (const locale of routing.locales) {
    if (pathname.startsWith(`/${locale}/`) || pathname === `/${locale}`) {
      return DISABLED_MESSAGES[locale] ?? DISABLED_MESSAGES.en
    }
  }
  return DISABLED_MESSAGES.en
}
```

API 응답 부분에서 사용:
```ts
return NextResponse.json(
  { error: disabledMessage(pathname) },
  { status: 404 },
)
```
